Accessible design guidelines

Designing a service to be accessible affords two primary benefits to the end user. Firstly it increases the ability to access the service for all users, not just those with disabilities. And secondly it can lead to an overall simplification of the design and structure of the content.

Consistency

The importance of consistency cannot be overstated. Consistency in layout, in theme, and in language allow the user to quickly comprehend what you are telling them. Context switching can be jarring even for users with the luxury of visual cues, but these sudden changes can be even more confusing for users with limited access.

Keeping layout consistent so that text is aligned evenly down the whole length of the page and calls to action are kept in logical locations such as left aligned, all aid users who have low visibility and are forced to explore the page to find what they need.

If the content jumps around the page unexpectedly, the user will be forced to do the same.

If the switch is required such as when transitioning from one discreet part of a service to another, a textual description of the change can help guide the user. Such as informing them that they have completed their triage.

If a layout requires information to be displayed in columns it can be useful to show visual clues that there is information off to the right of the screen by simply adding a horizontal rule.

Meta data

Language

The language attribute allows HTML parsers (such as screen readers) to understand what the intended language of the content is. This extra information can be used by those applications to tailor their experience for the end user. Such as more accurate announcement.

See the W3C for further information.

Title

Every page must have a title which describes the page concisely. More advanced users appreciate the title being prefixed with the site's name, allowing them to identify and group browser tabs.

General semantics

Headings

All pages must have a H1 tag. Also, appropriate use of header tags that follow the document structure help to navigate the page at a high level.

Users of screen reading software such as Apple's VoiceOver will more likely browse the site in group mode, which allows them to navigate the DOM at a much high level and therefore more quickly. For this to work well it is important to ensure consistency across HTML elements. So don't start by using links to describe a collection of elements and then switch to buttons. Even if that makes more visual sense to a sighted person.

Strong tag

Using the HTML strong tag to emphasis content will ensure that screen readers apply the correct emphasis to wording. The bold tag will often not by announced with extra emphasis.

Keyboard only users

Users of keyboards only will rely heavily on the tab key and the tab index of page elements. Ensure navigation can be completed intuitively with appropriate tab indexing of fields and elements.

Forms and interactions

If something changes when a user interacts with the page it is important to inform them of the change and place their focus where they would expect to now be placed. If a button shows some new content or form fields, it would be unexpected if their focus was suddenly placed beneath the new content.

One way to inform screen readers of the change is the aria-expanded attribute which will announce the change to supported agents.

Fields

Fields should be clearly marked as either optional or mandatory. If all fields are the same then the whole form can be marked accordingly.

Fieldsets and legends

Group related form fields into fieldsets and supply legends that describe the collection. If you don't want the legend text to be visible you can always hide it with styles.

Labels

Ensure all fields have relevant labels.

Try not to repeat information when using labels. For instance, don't ask the user for their full name and then label two text boxes as first name and last name. This could suggest two conflicting pieces of information to someone who can't see the layout of the page.

Validation

When validation is displayed after an interaction, it should be announced. Ideally as a summary. And focus should be placed accordingly.

Colour and contrast

Tools such as Colour Contrast Analyser can assist in identifying areas of the site which will impact users with colour contrast sensitivity. Ensuring that the mix of background colours and foreground colours have a high enough contrast ratio for legibility.

Full screen overlays

If the user is using a screen magnifier, a full screen overlay can create an abrupt switch in context forcing them to explore the page to understand what happened. Often it can be more immediately intuitive to simply up-date the button they just clicked with information about what is happening.

Additional suggestions

  • When designing pages it's important to remember - the goal of accessibility is to increase the ability for all users to complete what they need to when using the service.
  • Simply running a page through a screen reader will show up a great deal of issues early on in development.
  • Ensure all elements on a page are being announced fully and as intended. Especially telephone numbers.
